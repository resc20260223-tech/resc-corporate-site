# CLAUDE.md — RESC サンプルHP制作プロジェクト

> このファイルは Claude Code がこのプロジェクトで作業する際に毎回読み込む常設の指示書です。
> プロジェクトの目的・技術構成・作業の型・お作法をまとめています。作業前にここを前提として振る舞ってください。

## 1. プロジェクトの目的
株式会社RESC（代表：豊永光城／福岡）の、AI活用ホームページ制作サブスク事業のための**営業用サンプルサイト**を制作する。
- 顧客向けの訴求は「月4,980円〜、ホームページ担当者を1人雇う」。"AI"は前面に出さず、"専属担当者を雇う"フレーミングで統一する。
- サンプルは実在店舗ではなく架空店舗。各ページ冒頭に「これはRESCが作成したサンプルサイトです。実店舗ではありません。」の注記を必ず入れる。
- 8業種（cafe / salon / clinic / legal(士業) / build(工務店) / retail(小売) / nail / tech）のサンプルを、共通テンプレートから業種ごとに作り込む。

## 2. 技術スタックとフォルダ構成
- **バニラHTML / CSS / JS**（フレームワーク不使用）。CSS・JSは各HTMLファイル内に直接記述（インライン）。
- ホスティング：**GitHub Pages**（リポジトリ `resc20260223-tech/resc-corporate-site`、Public）。
- 公開URL：`https://resc20260223-tech.github.io/resc-corporate-site/`
- フォルダ構成：
  - `index.html` … コーポレートサイトのトップ（制作事例ギャラリーから各サンプルへリンク）
  - `samples/{業種}.html` … 業種別サンプルページ（例：`samples/cafe.html`, `samples/salon.html`）
  - `samples/images/{業種}/` … 業種別の画像（例：`samples/images/cafe/cafe_menu_latte.png`）
- 各業種HTMLは**同一テンプレートベース**。CSS変数（`--primary` 等）の値（色）だけ業種ごとに差し替えている。セクション構成は notice→hero→about→menu→gallery→contact→footer で共通。
- フォント：Noto Sans JP / Inter 系。
- 画像生成：**Genspark Claw**（`gsk img` CLIツール）。基本ルール＝英語プロンプト・人物や実在ロゴなし推奨・テキストなし。比率はヒーロー/ギャラリーは16:9、メニューカードは構図を揃えたい場合4:3も可。生成物は `~/Desktop/{業種}写真/` に保存し、後で `samples/images/{業種}/` にコピー。

## 3. ローカルプレビューの起動方法
- リポジトリのルートで簡易HTTPサーバーを立てる：`python3 -m http.server 8080`（ポートは空きに応じて8091等でも可）。
- ブラウザで `http://localhost:8080/samples/{業種}.html` を開いて確認する。
- 画像は相対パス参照のため、`samples/images/{業種}/` が正しい階層にあれば表示される。
- 注意：`.claude/launch.json`（ローカルサーバー設定）はコミット対象に含めない。

## 4. よくやる作業パターンと手順の型
1業種を「カフェと同じ4機能」に仕上げるのが基本パターン：
1. **画像生成**：Genspark Clawで、メニュー/サービス写真6枚＋雰囲気（ギャラリー）写真6枚を生成し `~/Desktop/{業種}写真/` に保存。
2. **現状確認**：Claude Codeで対象HTMLの構造を確認（cafe.htmlと同一テンプレートかも確認）。
3. **画像コピー**：`~/Desktop/{業種}写真/` の画像を `samples/images/{業種}/` にコピー。
4. **4機能を組み込み**（Claude Codeで一括指示）：
   - 写真付きメニューグリッド（`menu-card` 形式。カテゴリ分け、レスポンシブ、中央配置）
   - ヒーローアニメ（ケンバーンズ：背景ゆっくりズーム＆パン、ease-out、20s alternate infinite ＋ 文字フェードイン＋スライドアップ）
   - 横スライドギャラリー（自動カルーセル：右→左無限ループ、画像2セット並べて-50%移動、ホバーで一時停止、30〜40sで1周、overflow:hidden）
   - FAQチャットボット（右下フローティング、キーワードマッチ式、CSS＋バニラJS・API不使用、0.9秒後に自動オープン、×で閉じたら再オープンしない）
5. **ローカル確認** → **変更内容を報告してGOをもらう** → **Claude Codeでコミット＆プッシュ** → **本番URL（`?v=N`付き）で確認**。
- アクセシビリティ：全アニメに `prefers-reduced-motion: reduce` での無効化を必ず入れる。
- 業種の個性を足す場合あり（例：美容室ヒーローに「ハサミが1回チョキンと横切って画面が開く」オープニング演出）。凝った演出は「まず簡易版を試作→良ければ採用、ダメなら別案」で進める。

## 5. ハマった注意点・お作法
- **push忘れに注意（最重要）**：GitHub Desktopで「Commit」しただけでは本番に反映されない。必ず **「Push origin」** まで押す。上部が「Push origin ↑1」なら未push、「Fetch origin」なら push済み。「ローカルでは動くのに本番で動かない」時はまずここを疑う。
- **キャッシュ**：本番確認は必ずキャッシュ回避。URL末尾に毎回違う `?v=N` や `?nocache=N` を付け、`command + option + R`（スーパーリロード）で開く。
- **GitHub Pagesのビルド待ち**：push後、本番反映まで1〜3分かかる。すぐ反映されなくても数分待つ。
- **`.claude/launch.json` は除外**：ローカルサーバー設定ファイルなので、コミット時にチェックを外す。サイトには無関係。
- **`.DS_Store`**：Macが作る隠しファイル。サイトに影響なし。含めても実害はないが、気にするなら除外。
- **命名規則**：メニュー画像 `{業種}_menu_{品名}.png`、ギャラリー画像 `{業種}_gallery{1..6}.png`、ヒーロー `{業種}_hero.png`。既存の連番・命名に必ず揃える。
- **コミット単位**：1業種の刷新は1コミットにまとめる。メッセージ例「美容室サンプル刷新：写真付きメニュー・ハサミ演出ヒーロー・横スライドギャラリー・FAQチャットボット追加」。
- **コミット・push運用（2026年7月20日更新）**：旧ルール「Claude Code内でのpushはPAT認証で失敗しやすいためGitHub Desktop経由に限定」は撤回。**現在はClaude Codeがコミット・pushを担当する。**
  - pushの前に必ず以下を報告し、ユーザー（トミー）から「GO」「OK」等の承認を得てから実行する：①何を変更したか（要約）②コミット対象のファイル名（`git status`の結果）③主要な差分。
  - `git add`は変更したファイルを名指しで行う。`git add .`は使わない（`.gitignore`で`.DS_Store`／`.claude/`は除外済みだが、二重の安全策として）。
  - コミットメッセージは日本語で、後から見て変更内容が分かるように書く。
  - 承認が出るまではpushしない（commitまでは先に済ませておいてよい）。
  - **PAT等の機密情報はチャットに貼らない。**
- **既存を壊さない**：ある機能を足す/直す時、他セクションのHTML/CSS/JSに影響を与えないことを毎回指示に明記する。

## 6. イレギュラー対応で気をつけること
- **指示の取り違え**：「サイズを揃えて」等の曖昧な指示は、写真の"中身"か"配置(レイアウト)"かで意味が全く異なる。着手前に意図を1つ確認する。
- **`object-position` の限界**：写真の寄り感が揃わない場合、CSSの位置調整では直らない（元画像の構図差が原因）。その時は構図指定（medium shot / subject centered 等）を足して**画像を再生成**する。同名で上書きすればHTMLは触らず差し替え可能。
- **医療・接骨院系（clinic）のコピー**：断定的な効果効能表現（「必ず治る」等）は避ける。サンプルでも節度ある表現にする。
- **話が混線したら仕切り直す**：Claude Codeが別作業の質問に飛んだら、無関係な確認は「スキップ」し、やりたい作業だけを明確に単独で再指示する。
- **切り分けの順番**：本番で不具合が出たら「① push済みか（GitHub Desktopの表示）→ ② ビルド待ちか（数分待つ）→ ③ キャッシュか（nocache＋スーパーリロード）」の順で確認する。

## 7. 新デザイン標準（nail.html が第1号・以後の基準）
- **位置づけ**：`nail.html` で確立したデザインを新しい標準とする。今後の新規業種はこの標準で作る。既存業種（cafe/salon/clinic/professional/construction）は旧テンプレートのままで、いずれ引き上げる想定。
- **カラー設計**（業種ごとに色相は変えるが、構造は共通）：
  - メイン（濃/淡）、ベース（ホワイト/アイボリー/ソフト系）、アクセント（シャンパンゴールド `#D4AF37` 等）、テキスト用の濃色（コントラスト比4.5:1以上を必ず確保）
  - nail の例：`#D8B2B2` / `#E6C2C2` / `#FDFBF7` / `#D4AF37` / `#8B6363`（テキスト）
- **フォント**：和文＝明朝系（Shippori Mincho 等）、英数＝セリフ体（Cormorant Garamond 等）。見出しは letter-spacing 0.08〜0.15em、本文は 0.02em 程度。
- **レイアウト**：
  - Hero：背景写真＋ケンバーンズ＋文字フェードイン。帯（グラスモフィズム）は背景が見えなくなるため原則不使用。text-shadow か薄いオーバーレイで可読性を確保。
  - Concept：2カラム（写真＋テキスト）、スマホで1カラム縦積み。
  - Menu：3カラムのカード型（写真＋メニュー名＋キャッチコピー＋価格）。情報の優先順位は「メニュー名 ＞ 価格 ＞ キャッチコピー」。価格は埋もれさせない。
  - Gallery：横スライドカルーセル＋ホバーで一時停止＆scale(1.05)のZoom。
  - フッター：明るい背景＋ゴールドの細ボーダー枠＋ゴールド基調のボタン。
- **アニメーション**：Intersection Observer によるスクロールフェードイン（opacity 0→1、translateY 20px→0、0.8s ease-out、1回のみ、カードは0.1秒ずつstagger）。`prefers-reduced-motion` 対応必須。
- **チャットボット**：クイックリプライボタン（業種に応じた3択）を最初から表示。ゴールド細枠のピル型。
- **進め方**：一度に全部やらず、①デザインシステム（色・フォント）→②Hero→③Concept/Menu→④Gallery/フッター/アニメ の4ステップに分けて、1つずつ確認しながら進める。

## 8. 画像生成の環境変更
- Genspark Claw は解約。現在は **Gemini（Google AI Pro）** を使用。
- Geminiは会話文で指示 → 生成画像を手動ダウンロード → リネームして所定フォルダへ。
- **Geminiの制約（重要）**：
  - 生成物に透かし（スパークルマーク）が入る（Ultraプラン以外）。ヒーロー等の目立つ場所では要注意。
  - 参考画像に文字が入っていると生成物にも文字が焼き込まれる。焼き込まれた文字はCSS/JSでは消せない。
  - 微細なデザイン差（例：ワンカラー vs グラデーション）は描き分けが苦手。被写体そのものが明確に違うもの（手 vs 足 vs 施術シーン）で構成する方が確実。
  - 同じチャットで連投すると前の画像に引っ張られる。1枚ずつ、できれば新しいチャットで生成する。
- ファイル名は自動名（`Gemini_Generated_Image_xxx` / `1.png` 等）になるため、**生成順＝プロンプトの順**を利用してリネームすると早い（`stat` コマンドで秒単位のタイムスタンプを見れば正確な順序が分かる）。

## 9. 確認の鉄則（再掲・重要）
- ローカルも本番も、確認時は必ず URL末尾に `?nocache=N`（毎回違う数字）＋ `command+option+R`（スーパーリロード）。ローカルでもキャッシュは残る。
- 「ローカルで動くのに本番で動かない」時の切り分け順：①push済みか（GitHub Desktopの表示）→②GitHub Pagesのビルド待ち（1〜3分）→③キャッシュ。

## 12. 画像変換スクリプト（scripts/convert-images.sh）
- `scripts/convert-images.sh`：Gemini生成画像を一括処理する自動化スクリプト。
- 入力：`~/Desktop/gemini-input/{業種}/{hero,menu,gallery}/` にPNGを配置。
- 処理：hero=リサイズなし／menu=幅800px／gallery=幅1200px、いずれもWebP変換（品質82、RGBA→RGB、python3+Pillow使用）。
- 出力：`samples/images/{業種}/` に配置、既存の命名規則に準拠。
- 実行：`./scripts/convert-images.sh {業種名}`（1業種ずつ）。
- **用途**：7章の「残り5業種（cafe/salon/clinic/professional/construction）の新デザイン標準への引き上げ」の際に画像を差し替えるために使う想定。
- **現状**：5業種の刷新自体は、**ユーザーが今の画像に満足しているため保留中**。スクリプトは準備済みだが、着手時期は未定。
